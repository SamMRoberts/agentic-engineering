# Plugin Authoring Guide

Scope: this file applies to anything created or modified under `plugins/`. Read it alongside [../instructions/agentic-tooling/repo/AGENTS.md](../instructions/agentic-tooling/repo/AGENTS.md), which owns the broader principles (primitive choice, skill/agent authoring, MCP, validation, safety). This file is the concrete how-to for assembling a plugin in this repo.

## What lives in `plugins/`

- `plugins/<plugin-name>/` — one folder per installable plugin. Folder name is the plugin's stable id and must match every `name` field below.
- `plugins/marketplace.json` — generated by [../scripts/sync-marketplace.mjs](../scripts/sync-marketplace.mjs). **Do not hand-edit.** See "Marketplace integration" below.

Plugin folders use kebab-case names (`web-frontend-testing`, not `webFrontendTesting`).

## Required directory shape

```
plugins/<plugin-name>/
├── plugin.json                  # Copilot/Codex CLI manifest (entrypoint, agents, skills, mcpServers)
├── .codex-plugin/plugin.json    # Codex marketplace manifest (rich metadata, interface block)
├── .claude-plugin/plugin.json   # Claude Code manifest
├── .mcp.json                    # Optional: MCP servers this plugin depends on
├── .app.json                    # Optional: Codex Apps SDK manifest (often empty {})
├── hooks.json                   # Optional: lifecycle hook registrations
├── package.json                 # npm scripts: `test`, plugin-specific validators
├── package-lock.json            # Committed
├── README.md                    # User-facing usage, install, safety defaults
├── CHANGELOG.md                 # Semver entries
├── LICENSE
├── agents/                      # *.agent.md files
├── skills/                      # one subdir per skill, each with SKILL.md
├── hooks/                       # hook scripts (executable Node ESM)
├── lib/                         # shared validators, schema utils, yaml utils
├── schemas/                     # *.schema.yaml (JSON Schema draft 2020-12)
├── templates/                   # canonical artifact shapes
├── checklists/                  # human-readable safety / review checklists
├── profiles/                    # optional execution profiles
├── test/                        # node:test files; runs via `npm test`
└── mcp-app/                     # Optional MCP App subpackage (its own package.json)
```

Add only the folders you actually use. `web-frontend-testing` and `web-ux-testing` are reference implementations — copy from whichever matches your shape, but never copy logic without revisiting whether it applies.

## Plugin manifests — what goes where

Each host reads a different manifest. Keep them aligned but minimal:

- **`plugin.json` (root)** — Copilot and Codex CLI runtime. Contains `agents`, `skills`, `entrypointAgent`, `privateAgents`, optional `mcpServers`. No marketing metadata.
- **`.codex-plugin/plugin.json`** — what Codex marketplace UIs display. Contains `name`, `version`, `description`, `author`, `keywords`, `interface.*` (displayName, category, capabilities, defaultPrompt, brandColor). Optional `marketplace` override block (see below).
- **`.claude-plugin/plugin.json`** — Claude Code reads this. Pointers only: `agents`, `skills`, `hooks`, `mcpServers`. Claude resolves each pointer to a sibling file.
- **`.mcp.json`** — `mcpServers` map. Stub with `{}` if your plugin doesn't depend on any MCP server. Used by Claude (via the manifest pointer) and other hosts that respect it.
- **`hooks.json`** — `hooks` map keyed by lifecycle event (`PreToolUse`, `PostToolUse`, etc.). Empty `{}` is fine when there are no hooks.

`name` in every manifest must equal the folder name. Bump `version` (semver) in `package.json` and `.codex-plugin/plugin.json` together.

## Entrypoint agent

Every plugin must have exactly one user-invocable agent — the orchestrator. Stage work belongs in private subagents. The orchestrator's tool list should be `[read, search, todo, agent]` plus whatever the plugin specifically needs; it should not directly invoke browser tools, edit files, or run shell commands. See the orchestrator pattern at [web-frontend-testing/agents/web-frontend-testing.agent.md](web-frontend-testing/agents/web-frontend-testing.agent.md).

Record the orchestrator and private agents in `plugin.json`:

```jsonc
{
  "entrypointAgent": "my-plugin",
  "privateAgents": ["my-plugin-requirements", "my-plugin-plan", "my-plugin-execute"]
}
```

## Skills and their assets

Skills are mandatory for any repeatable step the orchestrator delegates. Each skill's `SKILL.md` owns the procedure; sibling folders own the data and code. The 200-line `SKILL.md` cap from the parent instructions is enforced — split into `templates/`, `checklists/`, `scripts/` before hitting it.

When a skill produces a structured artifact (plan, finding, report manifest), it must:

1. Reference a schema in `schemas/`.
2. Reference (and the agent must invoke) a validator script under the skill's `scripts/` that wraps shared logic in `lib/`.
3. Include at least one template that validates against the schema.

## Validators, lib, and tests

- Schemas: JSON Schema (draft 2020-12) authored as YAML in `schemas/`.
- Shared utilities: `lib/yaml-utils.mjs`, `lib/schema-utils.mjs`, `lib/<artifact>-lint.mjs`, `lib/<artifact>-validation.mjs`. Ajv + `ajv-formats` + `yaml`.
- Skill-level validators: thin wrappers under `skills/<skill>/scripts/validate-*.mjs` that exit `0` (pass), `1` (validation errors), `2` (bad usage/missing file).
- TypeScript consumers (e.g. an MCP App) need sibling `.d.mts` files next to the `.mjs` validators they import.
- Tests: `node:test` files under `test/`, run via `npm test`.

Use `WARN:` to stderr for non-blocking issues and `ERROR:` for blocking ones. Match the conventions in [web-frontend-testing/lib/plan-validation.mjs](web-frontend-testing/lib/plan-validation.mjs).

## Hooks

Hooks enforce what instructions only ask for. Three patterns this repo uses:

- `PreToolUse` denying file writes that contain credential-shaped literals.
- `PostToolUse` re-running a deterministic validator after a write to a known artifact path and returning `decision: block` with a `systemMessage` when errors are present.
- `PreToolUse` returning `permissionDecision: "ask"` for dangerous tools (e.g. `browser_evaluate`).

Keep hooks small, accept multiple input shapes (`tool_name` and `toolName`), and centralize shared parsing in `hooks/lib/hook-io.mjs`. See [web-frontend-testing/hooks/](web-frontend-testing/hooks/) for the pattern.

Register hooks in the plugin's `hooks.json` with `node hooks/<name>.mjs` commands and conservative timeouts (≤ 15s). Add hook tests under `test/` that spawn each hook with realistic stdin payloads.

## MCP apps (optional)

If the plugin ships an interactive UI, put it under `mcp-app/` as a self-contained subpackage with its own `package.json`, `vite.config.ts`, and `tsconfig.json`. Use `tsx` (not `bun`) for TS execution. Bundle the client with `vite-plugin-singlefile` so the resource is a single HTML file with no external assets. Register handlers before `app.connect()`. Always return a text `content` array alongside `structuredContent` so non-UI hosts still get usable output.

The MCP App should reuse the plugin's `lib/` validators, not duplicate them. See [web-frontend-testing/mcp-app/](web-frontend-testing/mcp-app/) for the reference layout.

## Marketplace integration

Marketplace listing is automatic — `plugins/marketplace.json` is generated from each plugin's manifests by [../scripts/sync-marketplace.mjs](../scripts/sync-marketplace.mjs).

- Pulled automatically: `name`, `category` (from `interface.category`), `description` (from `interface.shortDescription`).
- Defaults: `source: git-subdir`, `ref: main`, `policy: AVAILABLE / ON_INSTALL`.
- Per-plugin override: add a `marketplace` block to `.codex-plugin/plugin.json` (or root `plugin.json`) when defaults don't fit:

  ```jsonc
  {
    "marketplace": {
      "ref": "v0.2.0",
      "policy": { "installation": "PREVIEW", "authentication": "NONE" },
      "category": "Quality"
    }
  }
  ```

After any change to a plugin's manifest, run `node ../scripts/sync-marketplace.mjs` from the repo root (or rely on the pre-commit hook / CI auto-sync). Never edit `plugins/marketplace.json` directly — the next sync will overwrite the hand edit.

## Pre-merge checklist

Before opening a PR that adds or modifies a plugin:

1. `cd plugins/<plugin> && npm install && npm test` — must pass.
2. Run any plugin-specific validator (`npm run validate:plan`, etc.) against representative fixtures.
3. Smoke-test the entrypoint agent with at least one realistic prompt; verify the orchestrator stops on missing safety inputs rather than guessing.
4. If you added/changed an MCP server, MCP app, or hook, drive it end-to-end via `tsx` or a stdio harness.
5. From the repo root: `node scripts/sync-marketplace.mjs --check` exits 0.
6. Bump semver in both `package.json` and `.codex-plugin/plugin.json` and add a `CHANGELOG.md` entry. Major bump = breaking public interface (entrypoint agent name, skill name, schema field rename, tool shape).
7. Update the plugin's `README.md` if the user-facing surface changed.

## Common mistakes

- Editing `plugins/marketplace.json` by hand. Always re-run the sync script.
- Adding a destructive scenario, hook, or MCP tool without a `permissionDecision: ask` gate or explicit user confirmation path.
- Exposing private subagents as `user-invocable: true` for convenience. Keep the entrypoint surface to one.
- Letting the orchestrator hold any tool that mutates state directly. Mutations belong to the owning subagent.
- Duplicating validator logic in the MCP App instead of importing the plugin's `lib/`.
- Bumping a plugin's version without a `CHANGELOG.md` entry, or vice versa.
- Forgetting `ajv-formats` when the schema uses `format: uri`/`format: date-time`. Ajv silently passes unknown formats otherwise.
